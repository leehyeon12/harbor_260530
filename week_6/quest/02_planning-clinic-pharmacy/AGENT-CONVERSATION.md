## 1. app-mission-architect — 라운드 1 (발견 질문)

**입력한 아이디어(한 줄)**: "약국을 신규 개원하려는 사람이, 어디에 경쟁 약국이 적고 신규 개원하는
의료기관(잠재 고객)이 많은 좋은 입지가 있는지 지도로 한눈에 파악하게 해주는 웹앱을 만들고 싶다."

> 백지 상태로 discovery를 시키기 위해, 에이전트에게는 **기존 구현(임장 점수·기회 자리 등)을 알리지
> 않고** 아이디어 한 줄만 전달했다.

### 에이전트의 이해 요약
> 약국을 새로 열려는 사람이 "어디에 개원할까"를 고민할 때, 지도 위에서 두 가지 신호를 겹쳐 보여주는
> 웹앱. 경쟁 밀도(주변 기존 약국)와 수요 잠재력(신규 개원 의료기관)을 겹쳐, "경쟁은 적고 잠재 고객은
> 많은" 저평가된 입지를 찾아주는 입지 발굴 도구.

### 에이전트가 던진 질문 5개
1. 정확히 누구인가 — 첫 사용자 1명을 구체적으로(초보 약사 / 2호점 찾는 기존 원장 / 중개업자?)
2. 지금은 어떻게 하고 있나 — 이 앱 없이 지금은 어떻게 입지를 찾나, 가장 답답한 지점은?
3. "신규 개원 의료기관" 데이터를 어디서 얻나 — 실현 가능성의 급소
4. "좋은 입지"의 성공을 뭘로 재나 — 측정 가능한 성공 지표
5. 범위 밖으로 뺄 것 — v1에서 명확히 안 할 것

### 사용자 답변 (요약)
1. 첫 사용자 = 대학병원·동네약국 알바 경력의 **초보 약사**, 첫 개업 시점
2. 지금은 부동산 발품·지인 소개·지도앱 수기 확인. 최근 AI앱들은 오픈API 수준에 그침(차별점 존재)
3. 현재 공공데이터 사용, **향후 유료 데이터 확장 계획 → 비즈니스 모델 고려 필요**
4. "좋은 입지 추천 + 임장까지" 가 성공. 단 **리텐션 저하 우려**(수기 비교·임장은 앱 밖 일)
5. 원래 v1은 조회만 예정 → **로그인·즐겨찾기 추가**로 방향 전환(비즈니스 모델 포인트 겸용)

## 2. app-mission-architect — 라운드 2 (MISSION.md 초안)

사용자 답변 5개를 그대로 릴레이해 MISSION.md 초안 생성을 요청. 결과:

- 미션 문장, 문제 정의, 타겟 페르소나(김세연, 34세, 개업 준비 약사) 확정
- v1 In-Scope 4개(겹쳐보기 지도 / 스코어·추천 / 로그인+즐겨찾기 / 후보 비교), Anti-Scope 명시
- **리텐션 관점**을 별도 섹션으로 정면 대응 — "탐색→후보압축→임장→재검토"의 반복 여정으로 재정의
- 비즈니스 모델 방향(무료 검증 → 유료 데이터·과금 포인트) 한 단락
- 성공 지표 표(측정 가능한 5개 지표)
- **에이전트가 스스로 "확인 필요" 3가지를 남김**: ① 지역 범위(전국 vs 한 지역) ② "신규 개원"
  데이터 실현성 미확인 가정 ③ 성공 지표 수치의 임의성

## 3. 사용자 시나리오 테스트 (실제 앱 UI 기반)

에이전트 초안을 확정하기 전, **실제 index.html의 UI 라벨·로직**(임장 점수 계산식, 기회 자리 판정,
즐겨찾기 저장방식)을 직접 읽어 김세연 페르소나의 임장 여정 5단계(발견→스크리닝→임장 준비→현장→
재검토)를 시뮬레이션. 결과: **①②(발견·스크리닝)는 강함, ③④⑤(임장 실행·재검토 루프)는 완전히
비어 있음** → 리텐션 우려가 "기분 탓이 아니라 구조적 사실"임을 확인. 이 진단을 MISSION.md
"알려진 갭" 섹션에 반영.

## 4. 확인 사항 확정 (사용자 결정)

1. 지역 범위 → **당근마켓식 지역 지정 + 동(洞) 단위 세밀화**(구 단위에서 확장)
2. "신규 개원 의료기관" → 이미 `openDate` 필터로 **구현되어 있음** 확인 → v1 **핵심 기능으로 격상**
   (에이전트의 소극적 절충안 폐기)
3. 성공 지표 → 에이전트 제안값으로 확정

→ MISSION.md를 위 결정 반영해 최종본으로 갱신.

## 5. dev-kickstart 호출

MISSION.md 확정 후 개발계획 수립 요청. 이번엔 **실제 기존 스택 정보를 전달**(단일 index.html +
Vercel 서버리스 api/*.js 프록시, DB·Auth 없음, 즐겨찾기는 localStorage뿐) — 백지 스택 선택이 아니라
**브라운필드(기존 자산 위에 얹는) 계획**을 명시적으로 요청.

### 에이전트의 핵심 판단
- 표준 3옵션(단일파일/Supabase/Next.js)을 기계적으로 강요하지 않고, **"기존 Vercel 프록시 유지 +
  Supabase(Auth+DB) 얹기" 하이브리드**로 판단. 근거: 공공데이터 `SERVICE_KEY`는 서버 프록시 없이
  못 숨기고, 로그인·서버 즐겨찾기는 Supabase 없이 못 만듦 — 한쪽만으론 둘 다 불가능.
- Next.js 전환은 90KB index.html 재작성 리스크 대비 이득(SSR·SEO)이 핵심가치(지도 인터랙션)와
  무관하다고 판단해 **배제**.
- **결정적 발견**: 기존 코드의 `favoritesStore`가 이미 어댑터로 분리돼 있어(index.html L133~149),
  localStorage 구현체를 Supabase로 교체만 하면 서버 저장 완성 — 브라운필드 리스크를 크게 낮춤.
- 동(洞) 단위 세밀화는 공공데이터 API가 동 필터를 지원하지 않을 가능성이 높다고 진단(도로명주소
  본문에 동 이름 없음) → 좌표 기반 클라이언트 필터로 우회하는 리스크 분석과 Phase 배치.
- favorites 테이블 스키마를 컬럼 수준까지 제시(RLS 4정책 포함), 총 17개 TODO를 Phase 0~4로 구성.
- **결정 필요 3건**을 스스로 남김: 점수 저장(스냅샷 vs 재조회) / 비로그인 처리(필수 vs 마이그레이션)
  / 동 세밀화 1차 방식(반경선택 vs 행정동태깅).

## 6. 결정 필요 3건 확정 (사용자 결정)

1. 임장 점수 저장 → **스냅샷 저장** (비교뷰 안정성)
2. 비로그인 즐겨찾기 → **로그인 필수**(v1 단순화, 마이그레이션 로직 안 만듦)
3. 동 세밀화 1차 → **지도 반경 선택**(API 호출 부담 0, 안전)

→ DEV.md "확정된 결정" 섹션 + 관련 TODO·Open Questions를 이 결정으로 갱신해 최종본 완성.
